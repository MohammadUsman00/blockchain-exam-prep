Weeks 1–3: Cryptography, Bitcoin Basics, Merkle Trees
⚠️ Week 1–3 Pattern
Hash properties tested EVERY exam. Know: SHA-256, avalanche effect, public/private key roles, Merkle tree math, Bitcoin block structure.
W1·Q1HOT
Which statement correctly describes the SHA-256 hash function?
  • a) Output size grows with input size
  • b) Always produces a fixed 256-bit output regardless of input size
  • c) It encrypts data for secure communication
  • d) Different inputs can map to same output intentionally
Answer: (b)
SHA-256 always outputs exactly 256 bits. Critical: hashing ≠ encryption. Hashing is one-way; encryption is reversible with a key. This distinction is tested every semester.
W1·Q2HOT
The avalanche effect in hash functions means:
  • a) Hash function is slow to compute
  • b) A single-bit change in input causes ~50% of output bits to change
  • c) The hash avalanches when input is very large
  • d) Output bits are always 0 or 1 alternately
Answer: (b)
Avalanche effect: flip 1 input bit → approximately half (~128 of 256) output bits flip. This makes tiny tampering detectable.
W1·Q3HOTTRAP
Which property guarantees that given a hash output H, it is computationally infeasible to find the input M?
  • a) Pre-image resistance
  • b) Collision resistance
  • c) Second pre-image resistance
  • d) Avalanche resistance
Answer: (a)
Pre-image resistance: given H(m), cannot find m.
Second pre-image resistance: given m1, cannot find m2 where H(m1)=H(m2).
Collision resistance: cannot find any m1≠m2 where H(m1)=H(m2).
These are three DISTINCT properties — know all three!
W1·Q4CALC
A hash function produces 128-bit output. How many hash computations are needed on average for a birthday (collision) attack?
  • a) 2¹²⁸
  • b) 2⁶⁴
  • c) 2³²
  • d) 2¹⁶
Answer: (b) — Formula: 2^(N/2)
Birthday attack on N-bit hash requires ~2^(N/2) computations.
128-bit → 2^(128/2) = 2⁶⁴ computations.
W1·Q5HOT
Which provides non-repudiation?
  • a) Symmetric encryption
  • b) SHA-256 hash
  • c) Digital signature using private key
  • d) HMAC with shared secret
Answer: (c)
Non-repudiation = sender cannot deny sending. Only private key digital signature achieves this — only you have your private key, so only you could have signed.
W1·Q6TRAP
HMAC (Hash-based Message Authentication Code) provides:
  • a) Non-repudiation and authentication
  • b) Message integrity and authentication, but NOT non-repudiation
  • c) Confidentiality only
  • d) Digital signature equivalent
Answer: (b)
HMAC uses a shared secret key. Both parties know the key, so either could have generated it — no non-repudiation. Digital signatures use asymmetric keys, providing non-repudiation.
W2·Q1HOTTRAP
Bob receives Alice's message encrypted for confidentiality and signed for authenticity. Which keys does Bob use?
  • a) Alice's Public Key + Bob's Private Key
  • b) Bob's Private Key to decrypt + Alice's Public Key to verify signature
  • c) Bob's Public Key to decrypt + Alice's Private Key to verify
  • d) Shared symmetric key for both
Answer: (b) — Golden Rule
Encryption: Alice encrypts with Bob's PUBLIC key → Bob decrypts with Bob's PRIVATE key.
Signature: Alice signs with Alice's PRIVATE key → Bob verifies with Alice's PUBLIC key.
Memory trick: "Encrypt TO someone = use THEIR public key. Verify FROM someone = use THEIR public key."
W2·Q2HOT
Which statement about asymmetric vs symmetric encryption is TRUE?
  • a) Asymmetric is faster than symmetric encryption
  • b) Symmetric encryption enables non-repudiation
  • c) Asymmetric encryption is slower but solves the key distribution problem
  • d) Symmetric requires two different keys
Answer: (c)
Symmetric = one shared key, fast, but how do you securely share the key? Asymmetric = public/private pair, solves key distribution, but computationally expensive. In practice: hybrid (TLS uses asymmetric to exchange symmetric key).
W2·Q3DEFN
In a PKI (Public Key Infrastructure), what is the role of a Certificate Authority (CA)?
  • a) Mine blocks in proof-of-work
  • b) Issue digital certificates binding public keys to identities, and revoke them when needed
  • c) Store private keys for users
  • d) Validate smart contract execution
Answer: (b)
CA vouches that "this public key belongs to this person/organization" via a signed certificate. If a private key is compromised, CA can revoke the certificate. Hyperledger Fabric uses CA for MSP identity management.
W2·Q4HOT
What does the nonce in a Bitcoin block header do?
  • a) Identifies the miner who found the block
  • b) Stores the number of transactions
  • c) A 32-bit value miners vary to find a hash below the target difficulty
  • d) Encrypts the coinbase transaction
Answer: (c)
Nonce is a 32-bit field (0 to 2³²-1 ≈ 4.29 billion values). Miners try different nonce values until hash(block_header) < target. If all nonces exhausted, miner changes extra-nonce in coinbase or timestamp.
W2·Q5HOT
A Bitcoin block ID is computed as:
  • a) SHA-256 of the entire block (header + transactions)
  • b) Double SHA-256 of the block header only
  • c) SHA-256 of the Merkle root only
  • d) SHA-256 of the previous block hash
Answer: (b)
Block ID = SHA256(SHA256(block_header)). Header only, not entire block. Header contains: version, prev_block_hash, merkle_root, timestamp, bits (difficulty), nonce.
W3·Q1CALCHOT
Bitcoin block reward is 3.125 BTC, block time = 10 min. BTC mined in 24 hours?
  • a) 312.5 BTC
  • b) 450 BTC
  • c) 900 BTC
  • d) 225 BTC
Answer: (b)
Blocks/day = (24×60)/10 = 144 blocks/day
Total = 144 × 3.125 = 450 BTC/day
W3·Q2CALCHOT
Merkle tree with 16 transactions — total number of nodes (all levels including leaves and root)?
  • a) 32
  • b) 16
  • c) 31
  • d) 15
Answer: (c) — Formula: 2n−1
For a perfect binary tree with n leaves: Total nodes = 2n−1 = 2(16)−1 = 31
Breakdown: 16 leaves + 8 + 4 + 2 + 1 = 31
W3·Q3CALCHOT
To prove a transaction is in a Merkle tree with 64 transactions, how many hashes must be provided in the Merkle proof?
  • a) 32
  • b) 8
  • c) 6
  • d) 64
Answer: (c) — Formula: log₂(n)
Merkle proof path length = log₂(n) = log₂(64) = 6 hashes
Each level provides the sibling hash. SPV (Simplified Payment Verification) uses this.
W3·Q4TRAP
A Merkle tree has an odd number of transactions. What happens?
  • a) Extra empty transaction is added
  • b) The last transaction's hash is duplicated
  • c) The tree is rebuilt in a different structure
  • d) An error is thrown and no block is produced
Answer: (b)
Bitcoin handles odd leaves by duplicating the last hash. This allows the tree to always be a perfect binary tree. Note: This duplication behavior has been the source of a historical Bitcoin vulnerability (CVE-2012-2459).
W3·Q5HOT
Which halving event set Bitcoin's reward to 3.125 BTC?
  • a) 3rd Halving (2020)
  • b) 4th Halving (2024)
  • c) 2nd Halving (2016)
  • d) 5th Halving (2028)
Answer: (b)
Halving history: 2009=50 BTC → 2012=25 → 2016=12.5 → 2020=6.25 → 2024=3.125 BTC. Every 210,000 blocks (~4 years).
W3·Q6DEFN
What is a UTXO (Unspent Transaction Output)?
  • a) A mining reward given to miners
  • b) An unverified transaction waiting in mempool
  • c) A discrete chunk of Bitcoin that has been received but not yet spent, used as input to new transactions
  • d) An invalid transaction returned by the network
Answer: (c)
Bitcoin uses UTXO model (not account balances). Each transaction consumes UTXOs as inputs and creates new UTXOs as outputs. Your "balance" = sum of all your UTXOs. Ethereum uses account model instead.
W3·Q7HOT
What is the purpose of a coinbase transaction?
  • a) Transaction to buy crypto on Coinbase exchange
  • b) A transaction that reclaims lost Bitcoin
  • c) The first transaction in every block that creates new Bitcoin as block reward for the miner
  • d) A transaction signed by Satoshi Nakamoto
Answer: (c)
Coinbase transaction has no inputs (creates Bitcoin from nothing). It rewards the miner with block subsidy + transaction fees. The coinbase contains the "extra nonce" which miners use when 32-bit nonce space is exhausted.
W3·Q8CALC
Bitcoin difficulty is adjusted every 2016 blocks. If blocks were being mined at 8 minutes average (faster than 10 min target), what happens to difficulty?
  • a) Difficulty decreases
  • b) Difficulty increases (target hash value decreases)
  • c) Difficulty stays same — only adjusted monthly
  • d) Block time increases automatically to compensate
Answer: (b)
If blocks found faster than 10 min → too much hash power → increase difficulty (lower the target threshold). Formula: New_target = Old_target × (actual_time / 2016×10 minutes). Difficulty ∝ 1/target.
W3·Q9HOT
Which field in the Bitcoin block header links it to the previous block?
  • a) Merkle root
  • b) Nonce
  • c) Previous block hash
  • d) Timestamp
Answer: (c)
Each block header contains the hash of the previous block header. Changing any old block would change its hash, breaking the link chain — this is what makes blockchain tamper-evident.
W3·Q10HOT
What does a Bitcoin miner actually compete to find?
  • a) A prime number larger than the previous prime
  • b) The private key of another miner
  • c) A nonce such that SHA256(SHA256(block_header)) produces a hash below the target difficulty value
  • d) The longest possible transaction chain
Answer: (c)
PoW puzzle: find nonce N such that Hash(block) < Target. "Below target" means the hash starts with a certain number of leading zeros. Difficulty = how many leading zeros required.

Weeks 4–6: Forks, Ethereum, Consensus — BFT/Paxos
⚠️ Week 4–6 Pattern
Fork definitions, 51% attack limits, Ethereum gas/units, BFT formula (n≥3f+1 vs 2f+1) — all appear every exam.
W4·Q1HOTTRAP
Which can a 51% attacker NOT do?
  • a) Reverse their own recent transactions
  • b) Block specific transactions from confirming
  • c) Steal Bitcoin directly from other users' wallets
  • d) Mine blocks faster than the rest of the network
Answer: (c)
CAN: double-spend own txs, censor txs, mine competing chain, earn extra rewards.
CANNOT: steal from wallets, change max supply, reverse old deeply-confirmed blocks, break cryptography.
W4·Q2HOT
A soft fork is defined as:
  • a) A permanent chain split requiring all nodes to upgrade
  • b) A backward-compatible protocol upgrade where non-upgraded nodes still accept new blocks
  • c) A temporary split when two miners find blocks simultaneously
  • d) A fork introduced by an attacker to double spend
Answer: (b)
Soft fork: tightens rules, old nodes accept new blocks. Hard fork: loosens rules, old nodes reject new blocks → chain permanently splits. Examples: SegWit = soft fork. Bitcoin Cash = hard fork.
W4·Q3HOT
A temporary fork (natural fork) occurs when:
  • a) Two miners find valid blocks almost simultaneously, creating two competing chains
  • b) An attacker controls 51% of hash power
  • c) A software upgrade is applied to half the nodes
  • d) Network connectivity is permanently lost
Answer: (a)
Resolved by the longest chain rule (most cumulative PoW). The network converges when one chain gets another block first. This is why Bitcoin recommends 6 confirmations for finality.
W4·Q4DEFN
What is a Sybil attack and how does PoW prevent it?
  • a) Creating fake transactions — PoW prevents by signature verification
  • b) Creating many fake identities to gain network influence — PoW prevents by making identity creation computationally expensive
  • c) Flooding the network with spam — PoW prevents with transaction fees
  • d) Eavesdropping on transactions — PoW prevents via encryption
Answer: (b)
In Sybil attack, adversary creates thousands of fake nodes to control vote. PoW prevents this: influence ∝ computational power, not number of identities. Creating 1000 fake identities doesn't help without 1000× the hardware.
W4·Q5HOT
Bitcoin's "longest chain wins" rule actually means:
  • a) The chain with the most blocks (by count)
  • b) The chain with the most cumulative proof-of-work (most difficulty)
  • c) The chain with the largest transactions
  • d) The chain chosen by the node with most connections
Answer: (b)
"Longest" = most cumulative work, not most blocks. Two chains of same block count but different difficulties would resolve in favor of the higher-difficulty chain.
W5·Q1HOTCALC
Which Ethereum unit relationship is CORRECT?
  • a) 1 Ether = 10⁹ Wei
  • b) 1 Gwei = 10⁹ Wei
  • c) 1 Ether = 10⁶ Gwei
  • d) 1 Finney = 10⁹ Wei
Answer: (b)
1 Ether = 10¹⁸ Wei (NOT 10⁹ — common trap!)
1 Gwei = 10⁹ Wei ✓
1 Ether = 10⁹ Gwei
Gas prices specified in Gwei. Largest denomination = Ether.
W5·Q2HOT
Which Ethereum operation costs zero gas?
  • a) Calling a function that writes to storage
  • b) Reading/querying blockchain state via a view function
  • c) Emitting an event
  • d) Transferring Ether to another address
Answer: (b)
In Ethereum: read operations (view/pure functions) = free (0 gas) when called externally. State-modifying operations (write storage, transfer ETH, emit events) cost gas. This is because reads don't require consensus.
W5·Q3HOT
Which consensus mechanism uses Intel SGX trusted hardware for random wait time generation?
  • a) Proof of Stake
  • b) Proof of Burn
  • c) Proof of Elapsed Time (PoET)
  • d) Delegated Proof of Stake
Answer: (c)
PoET (used in Hyperledger Sawtooth): Each node gets a random wait time from trusted SGX enclave. Shortest wait time wins. Fair without computational waste. Verifiable because SGX produces signed proof.
W5·Q4DEFN
What is the EVM (Ethereum Virtual Machine) and why is it important?
  • a) A virtual mining rig that runs Ethereum node software
  • b) The hardware infrastructure for Ethereum data centers
  • c) A sandboxed, deterministic runtime environment that executes smart contracts identically on all nodes
  • d) An emulator for running Bitcoin scripts on Ethereum
Answer: (c)
EVM ensures deterministic execution: same code + same input = same output on ALL nodes worldwide. This is essential for consensus. EVM is Turing-complete but gas-bounded (prevents infinite loops).
W5·Q5DEFN
What is Proof of Burn in blockchain consensus?
  • a) Destroying hardware to prove commitment
  • b) Sending cryptocurrency to a verifiably unspendable address to gain mining rights proportional to burned amount
  • c) Burning electricity via PoW computation
  • d) A penalty mechanism for malicious validators in PoS
Answer: (b)
PoB: burn coins to unspendable "eater address" → gain virtual mining power proportional to burned coins. Demonstrates long-term commitment (cannot recover burned coins). Alternative to PoW's energy waste.
W6·Q1HOTCALC
A system has 7 nodes under Byzantine Fault Tolerance model. Max faulty nodes tolerated?
  • a) 1
  • b) 2
  • c) 3
  • d) 4
Answer: (b) — n ≥ 3f+1
7 ≥ 3f+1 → 3f ≤ 6 → f ≤ 2. Max 2 Byzantine faults.
Note: For crash faults (Paxos): n ≥ 2f+1 → 7 ≥ 2f+1 → f ≤ 3. Different answer!
W6·Q2CALC
What is the minimum number of nodes needed to tolerate 3 Byzantine faults?
  • a) 6
  • b) 7
  • c) 10
  • d) 9
Answer: (c) — n = 3f+1
n = 3(3)+1 = 10 nodes minimum for BFT with f=3.
For crash faults: n = 2(3)+1 = 7 nodes.
W6·Q3HOTTRAP
In Paxos, if fewer than a quorum of acceptors respond, which property is affected?
  • a) Safety — conflicting values may be decided
  • b) Liveness — no value can be decided, but no bad decisions are made
  • c) Both safety and liveness simultaneously
  • d) Neither — Paxos always proceeds
Answer: (b)
Safety = "nothing bad happens" (never violated in Paxos).
Liveness = "something good eventually happens" (violated if no quorum).
Without majority: cannot decide → liveness affected. But conflicting decisions never occur → safety preserved.
W6·Q4HOT
What problem does Multi-Paxos solve over basic Paxos?
  • a) Handles Byzantine faults unlike basic Paxos
  • b) Eliminates the repeated Prepare phase by electing a stable leader for multiple rounds
  • c) Reduces the number of nodes needed for consensus
  • d) Replaces Paxos entirely with a simpler algorithm
Answer: (b)
Basic Paxos: every instance requires full Prepare → Promise → Accept → Accepted (2 round-trips).
Multi-Paxos: elect stable leader once → only Accept → Accepted needed (1 round-trip) for subsequent slots.
W6·Q5HOT
Why does PBFT have O(n²) communication complexity?
  • a) Each replica must store O(n) copies of each message
  • b) In the Prepare and Commit phases, every replica must send messages to every other replica
  • c) The leader must do O(n²) cryptographic operations
  • d) View change requires O(n²) leader candidates
Answer: (b)
PBFT phases: Pre-prepare (1→n), Prepare (n→n²), Commit (n→n²). Each of n replicas messages all n replicas = n² messages. This is why PBFT doesn't scale beyond ~100 nodes.
W6·Q6HOT
PBFT requires n ≥ 3f+1. Why is 3f+1 specifically required (not 2f+1)?
  • a) To ensure at least f+1 honest nodes can form a quorum
  • b) To ensure quorum (2f+1) always contains at least f+1 honest nodes, even if f nodes are Byzantine and f are crashed
  • c) To handle network partitions with 3 segments
  • d) Because 3 is the minimum for triangle-based routing
Answer: (b)
With f Byzantine nodes: worst case = f faulty + f crashed/slow + (n-2f) honest respond. For quorum (2f+1) to have majority honest: n-2f > f → n > 3f → n ≥ 3f+1.

Weeks 7–9: Hyperledger, Algorand, Bitcoin-NG, ByzCoin
⚠️ Week 7–9 Pattern
Fabric transaction flow (Endorse→Order→Validate), MSP vs CA, Algorand VRF (j=0 meaning), Bitcoin-NG key-block vs micro-block all heavily tested.
W7·Q1HOT
Correct Hyperledger Fabric transaction flow order:
  • a) Order → Endorse → Validate
  • b) Endorse → Order → Validate
  • c) Validate → Endorse → Order
  • d) Endorse → Validate → Order
Answer: (b)
E-O-V:
1. Endorse: Client sends proposal to endorsing peers who simulate & sign.
2. Order: Orderer sequences transactions into blocks (no execution).
3. Validate: All peers check endorsement policy + MVCC (read-write set) → update world state.
W7·Q2HOTTRAP
What is the role of MSP (Membership Service Provider) in Hyperledger Fabric?
  • a) Issues X.509 certificates to participants (that's the CA)
  • b) Defines which identities are trusted and maps certificates to roles (admin, peer, client)
  • c) Executes chaincode
  • d) Orders transactions into blocks
Answer: (b)
MSP ≠ CA. CA issues certificates. MSP uses those certificates to determine membership and roles. MSP defines: which CAs are trusted, which certificate attributes map to which roles, revocation lists.
W7·Q3HOT
What is the purpose of Channels in Hyperledger Fabric?
  • a) To increase block production speed
  • b) To create logically separate ledgers among specific subsets of participants with data isolation
  • c) To replace the consensus mechanism
  • d) To store private keys securely
Answer: (b)
Channels = private "subnets" within Fabric network. Each channel has its own ledger, chaincode, and ordering service. Organizations not on a channel cannot see its transactions. Like private chat rooms within a larger platform.
W7·Q4HOT
In Fabric, the Orderer service's role is:
  • a) Verify signatures and execute chaincode
  • b) Store the world state database
  • c) Sequence transactions into blocks (does NOT validate/execute transactions)
  • d) Issue membership certificates
Answer: (c)
Orderer only sequences — it creates a deterministic order of transactions. It does NOT execute chaincode or validate business logic. After ordering, peers do the validation (endorsement policy check + MVCC).
W7·Q5HOT
Hyperledger Fabric's world state vs blockchain:
  • a) World state stores transaction history; blockchain stores current values
  • b) World state stores current key-value snapshot (fast reads); blockchain stores full transaction history (immutable audit log)
  • c) They store identical data in different formats
  • d) World state is encrypted; blockchain is plaintext
Answer: (b)
World state (CouchDB/LevelDB) = current state of all assets, queryable. Blockchain = append-only log of all transactions. World state can be rebuilt from blockchain. CouchDB enables rich JSON queries; LevelDB only simple key lookups.
W7·Q6HOT
Which statement about Hyperledger Fabric is FALSE?
  • a) Fabric uses modular design with pluggable consensus
  • b) Chaincode is Fabric's smart contract (written in Go/Java/Node.js)
  • c) Fabric requires a native cryptocurrency to pay for transactions
  • d) Fabric supports private data collections for confidentiality
Answer: (c) — FALSE
Fabric does NOT require a native cryptocurrency. This is a key difference from Ethereum. Fabric uses identity and policy-based access control instead of economic incentives.
W8·Q1HOT
In Bitcoin-NG, what are the two types of blocks and their roles?
  • a) Leader-blocks (elect leader) and transaction-blocks (transactions) — both mined via PoW
  • b) Key-blocks (PoW, elect epoch leader) and micro-blocks (no PoW, contain transactions, produced by leader)
  • c) Mining-blocks (for miners) and user-blocks (for users)
  • d) Checkpoint blocks (finality) and normal blocks (transactions)
Answer: (b)
Bitcoin-NG separates leader election from transaction processing:
Key-block: PoW puzzle → winner becomes epoch leader (contains public key).
Micro-blocks: Leader signs and broadcasts these rapidly (~every few seconds) with transactions. No PoW needed.
W8·Q2HOT
What problem does ByzCoin solve that Bitcoin-NG has?
  • a) Bitcoin-NG lacks transaction privacy
  • b) Bitcoin-NG's faulty leader can produce invalid micro-blocks with no accountability — ByzCoin adds PBFT-like collective signing for immediate finality
  • c) Bitcoin-NG is too slow — ByzCoin increases block size
  • d) Bitcoin-NG has no Merkle tree structure
Answer: (b)
Bitcoin-NG: faulty leader can create bad micro-blocks, detected only when next leader creates key-block. ByzCoin: uses CoSi (collective signing) — a committee of recent key-block winners signs each micro-block → immediate Byzantine fault tolerance.
W8·Q3HOT
CoSi (Collective Signing) reduces PBFT communication from O(n²) to:
  • a) O(n)
  • b) O(n log n)
  • c) O(n) messages and O(log n) latency via tree-based multi-signature aggregation
  • d) O(1) — constant time
Answer: (c)
CoSi uses a tree structure: leader broadcasts to tree, signatures aggregate up the tree (Schnorr signatures are aggregatable). Result: O(n) messages total, O(log n) communication rounds. Verifier sees one aggregated signature.
W9·Q1HOT
In Algorand's VRF cryptographic sortition, if a node receives j = 0, it means:
  • a) Node is selected as the block proposer
  • b) Node is NOT part of the consensus committee for this round
  • c) Node has voting weight of 0 in committee
  • d) VRF seed was invalid
Answer: (b)
Algorand VRF outputs: (hash, proof, j). j=0 → not in committee. j>0 → in committee with voting weight j. Nodes only reveal their role when they broadcast — prevents targeted attacks before selection is known.
W9·Q2HOT
Algorand's security assumption tolerates up to what fraction of malicious stake?
  • a) 33% (1/3)
  • b) 50% (1/2)
  • c) Less than 1/3 of total stake (similar to BFT)
  • d) Any amount — Algorand has no Byzantine assumption
Answer: (c)
Algorand assumes honest stake > 2/3 of total stake. Adversary can control <1/3 of stake. This maps to classic BFT assumption. Difference: Algorand achieves this at scale through VRF-based random selection.
W9·Q3HOT
In Hyperledger Indy, who are Stewards?
  • a) Regular users who hold credentials
  • b) Organizations that issue verifiable credentials (like universities)
  • c) Trusted institutions that run RBFT validator nodes on the Indy network and can approve Trust Anchors
  • d) Developers who write Indy chaincode
Answer: (c)
Hierarchy: Stewards (run validators, most trusted) → Trust Anchors (approved by Stewards, issue credentials, e.g., universities) → Users/Holders (receive credentials). Stewards ≠ Trust Anchors.
W9·Q4HOT
What is the DID (Decentralized Identifier) URI structure?
  • a) did:method:method-specific-identifier (e.g., did:sov:WRfXPg8dantKVubE3HX8pw)
  • b) did://method/identifier
  • c) method:did:identifier
  • d) identifier.method.did
Answer: (a)
DID format: did:[method]:[method-specific-id]
"did" = scheme, "method" = which DID system (sov=Sovrin/Indy, eth=Ethereum, etc.), last part = unique ID on that ledger.
Resolves to a DID Document containing public keys and service endpoints.
W9·Q5HOT
In Self-Sovereign Identity (SSI), the trust triangle involves:
  • a) Miner, Node, User
  • b) Issuer (issues credential), Holder (stores credential), Verifier (checks credential)
  • c) Admin, Peer, Client
  • d) Steward, Trust Anchor, Validator
Answer: (b)
Issuer (e.g., university) → issues VC to Holder (student) → Holder presents to Verifier (employer). Verifier checks on ledger (Indy) that issuer's schema/definition is valid. Issuer never involved in presentation.
W9·Q6DEFN
What does support_revocation: False mean in a Hyperledger Indy credential definition?
  • a) The credential can never be issued
  • b) The credential definition does not support revocation — once issued, credentials with this definition cannot be revoked
  • c) The credential is automatically revoked after issue
  • d) Revocation requires manual intervention
Answer: (b)
Setting support_revocation=False when creating a credential definition means the resulting credentials cannot be revoked later. Once set, cannot be changed. For revocable credentials, support_revocation=True and a revocation registry must be created.
W9·ByzCoin·Q1HOT
ByzCoin improves upon PBFT by integrating PoW and CoSi. Which correctly describes improvements for n nodes?
MSQ — select all correct
  • a) ByzCoin reduces communication from O(n²) to O(n) — WRONG (reduces to O(n) messages but latency is O(log n))
  • b) ByzCoin reduces consensus latency from O(n²) to O(log n) via tree-based structure
  • c) ByzCoin reduces verification from O(n) to O(1) via Schnorr multi-signatures
  • d) ByzCoin eliminates need for leader node — WRONG (still needs leader)
Answer: (b) and (c)
PBFT = O(n²) messages. ByzCoin CoSi tree = O(n) total messages, O(log n) latency. Schnorr multi-sig aggregates n signatures into 1 → O(1) verification for any third party. Leader still exists — just changes each key-block round.
W9·SSO·Q1TRAP
Which is NOT true about Single Sign-On (SSO)?
  • a) A single user can access multiple services with one login
  • b) A single identity provider typically maintains the identity
  • c) During authentication, identity is not directly exposed to all services
  • d) Identity holders use the Identity Provider to authenticate other identity holders
Answer: (d) — NOT TRUE

W7·BYSANTIUM·Q1HOT
What conditions must hold in a Byzantine quorum system to ensure safety?

W10·Q1HOT
HTLC (Hash Timelock Contract) — what happens if the secret is NOT revealed before timeout?
  • a) Funds are permanently locked
  • b) Funds auto-transferred to recipient after extra delay
  • c) Sender gets a refund — funds returned to original sender
  • d) Contract is re-executed with new parameters
Answer: (c)
HTLC two paths: (1) Reveal correct hash preimage (secret) before timeout → receiver gets funds. (2) Timeout expires without secret → sender gets refund. Neither party loses — either swap completes or reverts. Basis of Lightning Network and atomic cross-chain swaps.
W10·Q2HOT
In a Lightning Network payment channel, what happens on-chain vs off-chain?
  • a) All payments recorded on blockchain for auditability
  • b) Only channel open (funding) and channel close (settlement) are on-chain; intermediate payments are off-chain
  • c) Every 100 payments, a summary is recorded on-chain
  • d) Lightning only works for Bitcoin amounts over 0.1 BTC
Answer: (b)
Lightning Network: open channel (1 on-chain tx) → unlimited instant off-chain payments → close channel (1 on-chain tx settling final balance). Scales Bitcoin by orders of magnitude. Uses HTLCs for routing multi-hop payments.
W10·Q3HOT
What makes HTLC atomic in cross-chain swaps?
  • a) Both swaps happen in the same block
  • b) Either both transfers complete (secret revealed) or both revert (timeout) — no partial state possible
  • c) A trusted escrow atomizes the swap
  • d) Both chains use the same consensus protocol
Answer: (b)
Atomicity: Alice locks BTC with hash H, Bob locks ETH with same hash H. If Alice reveals secret S (where H=SHA256(S)), Bob's ETH is released. If not, both revert. Cannot have half-swap. No trusted third party needed.

📌 Consensus-Driven Data Transfer — Why Important?
In permissioned blockchain interoperability (e.g., Fabric to Fabric), the destination network must receive data that is provably finalized on the source. "Consensus-driven" means: data transfer happens only after the required endorsers (2f+1) have signed off. This prevents acting on data that might be "rolled back" or "orphaned." Contrast with permissionless: you'd use SPV (Merkle proofs + sufficient confirmations) to prove a tx is finalized on Bitcoin/Ethereum before bridging assets.
📌 Oracle Problem — Why Blockchain Cannot Verify Physical World
Blockchain is deterministic — all nodes must reach the same conclusion from the same data. But physical world data (temperature, product quality, sports scores) requires external input. An oracle is a trusted data feed. The oracle problem: if the oracle lies or is compromised, the blockchain faithfully records wrong data (immutably!). Solutions: Chainlink (decentralized oracles), multiple independent oracles with median value, IoT with hardware attestation.

All 9 Attack Types — Complete Reference
Week 11 had 5+ attack questions
Know mechanism, not just name. "What can they do" AND "what can they NOT do".
51% Attack
Attacker controls >50% hash power. Controls block production.
CAN: double-spend, censor txs, mine faster
CANNOT: steal wallets, change supply, break crypto
Sybil Attack
Create many fake identities to gain disproportionate influence in P2P network.
Defense: PoW (cost per identity), PKI (CA-verified identity)
Eclipse Attack
Fill victim's peer connection tables with attacker-controlled IPs to isolate them from honest network.
Splits honest network → attacker has majority over each isolated group. Defense: IP diversity limits.
Selfish Mining
Mine blocks privately, release strategically to waste honest miners' effort and earn disproportionate rewards.
Requires ~33% hashrate to be profitable. Orphans honest miners' blocks.
Double Spend
Spend same UTXO in two transactions. Use 51% to get conflicting tx confirmed.
Vendor ships goods → payment tx orphaned → attacker keeps goods + coins. 6 confirmations mitigate.
Transaction Privacy Leakage
Graph analysis of public blockchain reveals sender/receiver identity despite pseudonymity.
Defense: Chaff coins (mixins), CoinJoin, stealth addresses, Zero-Knowledge Proofs
Replay Attack
Rebroadcast valid signed transaction to execute it again (or on another chain after fork).
Defense: Transaction nonce (Ethereum), replay protection in hard forks, chain ID
Front-Running (DeFi)
Miner/validator sees pending tx in mempool, inserts own tx before it to profit (e.g., copy trade).
Possible because miners choose tx ordering. MEV (Miner Extractable Value). Defense: commit-reveal, private mempools.
Fomo3D Block Stuffing
Attacker wins prize by preventing other ticket purchases — stuffs blocks with high-fee useless transactions.
3 min remaining ÷ 30s/purchase = 6 prevented purchases. Timer expires → attacker wins pot.
Deep Dive: Eclipse Attack (Appeared 4× in W11)
Steps: (1) Attacker sends many ADDR messages with its IPs. (2) Victim's "new" table fills with attacker IPs. (3) Victim restarts (or attacker forces disconnect). (4) All reconnections go to attacker nodes. (5) Victim is now isolated — sees only attacker-controlled chain. Effect with 40%/30%/30% split: Attacker doesn't gain victim's power. Instead: victim faces attacker alone (30% vs 40%), rest of network faces attacker alone (30% vs 40%). Attacker has advantage over each isolated group.

  • c) To prevent Bitcoin network from crashing due to volume

W12·GOVT·Q1HOT

  • a) It speeds up access to large files
  • b) It allows deletion of stored data
  • c) It minimizes breach risks while keeping transparent record of who accessed what
  • d) It removes need for centralized storage entirely

Storing actual sensitive data on-chain is risky (immutable = can never be deleted, visible to all nodes). Storing only access logs (who/when/what accessed) keeps sensitive data off-chain while providing transparent, tamper-evident audit trail.
